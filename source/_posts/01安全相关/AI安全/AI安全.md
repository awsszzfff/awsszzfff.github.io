## 攻击面

### 模型权重

对抗性微调、重复查询提取模型权重，恶意替换权重的供应链攻击

### 训练数据

数据投毒（对抗性样本、后门触发器）

### API 与推理端点

接口验证，身份验证绕过、速率限制规避、参数操纵以及通过系统查询进行的模型反转攻击影响

### 系统提示词

提示词注入提取、上下文操纵绕过，或操作员提示词本身由用户影响的数据组装则被投毒

### 输出处理器

eg：输出呈现 HTML 时受到 XSS，输出驱动 shell 执行，受命令注入，输出插入查询受 sql 注入

### 插件与工具

智能体被劫持，对应的工具也受操纵（发邮件、修改文件、API 调用等）

### 向量数据库/RAG

对应文档植入恶意内容，提示词注入

### 编排层

链接 LLM 调用、管理内存并在智能体之间路由的框架，LangChain、LlamaIndex、AutoGPT 等，多智能体管道创造信任边界混淆和递归注入机会。

### 智能体记忆

记忆被投毒，造成持久的、系统级后门效应

### 部署基础设置

云托管、K8s、GPU 节点和模型服务基础设施，容器逃逸、凭据窃取、错误配置的 IAM，导致模型泄露

### AI 攻击链

每个组件看似正常，但组合起来被攻击者利用。

网页浏览客服聊天机器人，攻击者在公开博客文章中发表不可见字符提示词注入载荷，系统检索被投毒的文章并将其作为上下文提供给 LLM。LLM  处理指示其通过响应中精心制作的超链接将对话历史泄露到攻击者控制的 URL。如果前端将链接呈现为可点击的 HTML 而不进行净化，用户的浏览器在页面加载时静默请求攻击者的 URL。

### AI 架构中的信任边界

LLM 上下文窗口，系统提示词、对话历史、检索文档、工具输出、用户消息，都由同一模型使用同一机制处理。模型没有内在方式区分“来自操作员的指令”和“来自用户要求我摘要的文档的文本”。能够将文本注入上下文窗口的攻击者可能潜在劫持模型的指令遵循行为。这是提示注入漏洞类的根本原因,它源于基本的架构限制而非实现错误。
